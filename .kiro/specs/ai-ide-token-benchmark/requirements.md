# Requirements Document

## Introduction

The AI IDE Token Benchmark tool measures and compares usage in Kiro IDE under two conditions: a **baseline** condition (Kiro with no Power active — Kiro sends full file contents to Claude) and a **treatment** condition (Kiro with the **Context Lens Power** active — Kiro uses a compact signature map plus selective code expansion). The goal is to validate or challenge the Context Lens Power's claimed 10–15x reduction by measuring the actual usage delta across real coding sessions on a real target codebase.

The tool works by running a transparent HTTPS proxy (mitmproxy) that intercepts Kiro's calls to the AWS backend API and extracts credit usage data from each response's event stream. Since Kiro routes traffic through AWS (`generateAssistantResponse`) rather than directly to Anthropic, raw `input_tokens`/`output_tokens` are not exposed — instead, the proxy captures the per-request **credit usage** value, which is proportional to token consumption. The user follows a generated session script, manually entering prompts into Kiro's chat UI in two sequential runs (baseline then treatment). After both runs complete, the tool produces a comparison report showing per-session credit usage, reduction ratios, and an assessment of the actual reduction against the claimed 10–15x target.

### Context Lens Power

The Context Lens Power is a Kiro Power that:

- Indexes a codebase into a local SQLite database, capturing function and class signatures plus call edges.
- When Kiro answers a coding question, sends a compact signature map (~8k tokens) and expands only the relevant code units (~3k tokens) instead of sending full file contents.
- Claims a 10–15x token reduction compared to naive full-file inclusion.
- Is distributed as a Kiro Power with an auto-included steering file that triggers automatically when active.

## Glossary

- **Benchmark_Runner**: The core component that orchestrates proxy setup, session script generation, run collection, and report production.
- **Proxy**: The mitmproxy-based HTTPS interceptor that captures Claude API responses and extracts token usage data without modifying traffic.
- **Session**: An independent coding task conversation consisting of 5–10 turns, representing a coherent unit of programmer chat behavior on the target codebase (e.g., "fix the off-by-one error in pagination", "add input validation to the user registration endpoint").
- **Session_Script**: A structured document generated by the Benchmark_Runner that the user follows while manually entering prompts into Kiro's chat UI. Contains all sessions and their ordered prompts for both runs.
- **Turn**: One prompt-response pair within a Session.
- **Usage_Count**: The credit usage consumed in a Turn or Session, as reported by the AWS backend event stream. Credit usage is proportional to token consumption but is not a raw token count. Stored as `input_tokens` (credit × 1000, integer) and `output_tokens` (0) for compatibility with the reporting pipeline.
- **Token_Report**: The structured output produced after a benchmark run, containing per-turn and per-session Usage_Count data. Named "Token_Report" for historical reasons; the metric is credit usage, not raw tokens.
- **Baseline_Run**: A benchmark run with Kiro and no Power active — Kiro sends full file contents to Claude (Condition A).
- **Treatment_Run**: A benchmark run with Kiro and the Context Lens Power active — Kiro uses signature maps and selective code expansion (Condition B).
- **Comparison_Report**: The output document comparing Baseline_Run and Treatment_Run token counts, showing per-session reduction ratios, deltas, and an assessment against the claimed 10–15x reduction target.
- **Reduction_Ratio**: The ratio of Baseline_Run credit usage to Treatment_Run credit usage for a Session or aggregate run (Baseline / Treatment). A ratio of 10 means the treatment consumed 10x fewer credits (proportional to tokens).
- **Prompt_File**: A file containing the Session_Script in a defined format (JSON or YAML).
- **Config_File**: A file specifying run parameters such as output directory, output format, proxy port, and the path to the target repository.
- **Session_Script_Parser**: The component responsible for parsing and serializing Session_Script files.
- **Config_File_Parser**: The component responsible for parsing and serializing Config_File files.
- **Target_Repository**: The real codebase under test, specified via the `repo_path` Config_File parameter, against which all Session prompts are written.
- **MCP_Tool_Log**: A per-turn record of which Context Lens MCP tools (`context_index`, `context_query`, `context_expand`) were invoked by Kiro during that Turn, captured by the Proxy from intercepted request traffic.

## Requirements

### Requirement 1: Session Script Generation

**User Story:** As a benchmark operator, I want the tool to generate a session script of realistic coding tasks on my target codebase, so that I have a consistent, repeatable set of prompts that reflect how Context Lens is actually used.

#### Acceptance Criteria

1. THE Benchmark_Runner SHALL generate a Session_Script by loading the 10 issues defined in `example_project/ISSUES.md`, using each issue as the source of one Session.
2. WHEN loading an issue from `example_project/ISSUES.md`, THE Benchmark_Runner SHALL expand it into a 4-Turn multi-turn conversation: (1) a task description turn using the issue's prompt, (2) a clarifying question turn, (3) an implementation turn requesting the specific code change, and (4) a follow-up/verification turn confirming the change is correct.
3. THE Benchmark_Runner SHALL label each Session as single-file or cross-file based on the number of files listed in the issue's **Files** field: a single listed file produces a single-file label; two or more listed files produce a cross-file label.
4. THE Session_Script SHALL include a mix of single-file tasks (where lower token reduction is expected) and cross-file tasks (where higher token reduction is expected, as Context Lens is designed to shine).
5. THE Benchmark_Runner SHALL label each Session with its task type (single-file or cross-file) in the Session_Script.
6. THE Benchmark_Runner SHALL write the generated Session_Script to a Prompt_File in the output directory.
7. THE Session_Script_Parser SHALL parse a Prompt_File then serialize it back to an equivalent representation such that re-parsing the serialized form produces an identical Session_Script (round-trip property).
8. IF the output directory does not exist, THEN THE Benchmark_Runner SHALL create it before writing the Prompt_File.
9. WHERE the Prompt_File format is JSON, THE Benchmark_Runner SHALL validate the file against the defined JSON schema before use.
10. WHERE the Prompt_File format is YAML, THE Benchmark_Runner SHALL validate the file against the defined YAML schema before use.

### Requirement 2: Proxy Interception

**User Story:** As a benchmark operator, I want a transparent proxy to capture usage data from Kiro's API calls, so that I get credit usage metrics without modifying Kiro's configuration.

#### Acceptance Criteria

1. THE Proxy SHALL intercept HTTPS traffic between Kiro and the AWS backend API by running on localhost at a configurable port.
2. WHEN the Proxy intercepts an AWS `generateAssistantResponse` response, THE Proxy SHALL parse the Amazon Event Stream binary format and extract the `usage` (credit) value from the final event.
3. THE Proxy SHALL log the extracted credit usage data for each intercepted response, tagged with a timestamp.
4. THE Proxy SHALL pass all intercepted traffic through unmodified — it SHALL NOT alter request or response content.
5. WHEN the Proxy starts, THE Benchmark_Runner SHALL print instructions to the user for configuring the system proxy to route through the Proxy at localhost.
6. IF a response does not contain a credit usage field, THEN THE Proxy SHALL log a warning and continue capturing subsequent responses.
7. WHEN the Proxy stops, THE Benchmark_Runner SHALL write all captured usage data to a Token_Report file.
8. WHEN the Proxy intercepts a request, THE Proxy SHALL inspect the request body for MCP tool invocations and log an MCP_Tool_Log entry per Turn recording which Context Lens MCP tools (`context_index`, `context_query`, `context_expand`) were called, if any.

### Requirement 3: Run Orchestration

**User Story:** As a benchmark operator, I want the tool to guide me through both the baseline and treatment runs in a single invocation, so that I can complete the full A/B comparison without manual coordination.

#### Acceptance Criteria

1. THE Benchmark_Runner SHALL execute both the Baseline_Run and the Treatment_Run within a single invocation.
2. WHEN starting the Baseline_Run, THE Benchmark_Runner SHALL instruct the user to ensure no Power is active in Kiro (so that Kiro sends full file contents to Claude), then wait for the user to confirm readiness before proceeding.
3. WHEN starting the Treatment_Run, THE Benchmark_Runner SHALL instruct the user to activate the Context Lens Power in Kiro, then wait for the user to confirm readiness before proceeding.
4. DURING each run, THE Benchmark_Runner SHALL display the current Session and Turn from the Session_Script so the user knows which prompt to enter next in Kiro's chat UI.
5. WHEN the user signals that a Turn is complete, THE Benchmark_Runner SHALL advance to the next Turn in the Session_Script.
6. WHEN the user signals that a Session is complete, THE Benchmark_Runner SHALL instruct the user to start a new conversation in Kiro before beginning the next Session.
7. WHEN both runs are complete, THE Benchmark_Runner SHALL generate the Comparison_Report.

---

### Requirement 4: Usage Measurement

**User Story:** As a benchmark operator, I want credit usage recorded for each turn and session, so that I can make accurate proportional comparisons between runs.

#### Acceptance Criteria

1. WHEN the Proxy captures an API response with credit usage data, THE Benchmark_Runner SHALL record the credit usage value for that Turn. The credit value is scaled (×1000) and stored as `input_tokens` for compatibility with the reporting pipeline, with `output_tokens` set to 0.
2. THE Benchmark_Runner SHALL compute the aggregate usage for each Session as the sum of all Turn usage values within that Session.
3. THE Benchmark_Runner SHALL compute the aggregate usage for a run as the sum of all Session usage values within that run.
4. THE Benchmark_Runner SHALL record usage using only values captured from the API response event stream — it SHALL NOT use tokenizer-based estimation.
5. FOR ALL Sessions, the sum of Turn usage values SHALL equal the Session's aggregate usage (aggregation correctness property).
6. FOR ALL runs, the sum of Session aggregate usage values SHALL equal the run's total usage (aggregation correctness property).

---

### Requirement 5: Results Storage

**User Story:** As a benchmark operator, I want results saved to a file, so that I can review and share them after the run completes.

#### Acceptance Criteria

1. WHEN a run completes, THE Benchmark_Runner SHALL write the Token_Report to an output file in the directory specified by the Config_File.
2. THE Token_Report SHALL include, for each run: the run type (Baseline or Treatment), per-turn Token_Counts (input, output, total), per-session aggregate Token_Counts, and the overall aggregate Token_Count.
3. THE Token_Report SHALL include the timestamp of the run, the Prompt_File path, and the run type.
4. WHERE the output format is configured as JSON, THE Benchmark_Runner SHALL write the Token_Report as a valid JSON document.
5. WHERE the output format is configured as CSV, THE Benchmark_Runner SHALL write the Token_Report as a valid CSV document with a header row.
6. IF the output directory does not exist, THEN THE Benchmark_Runner SHALL create it before writing the Token_Report.
7. IF the output file cannot be written, THEN THE Benchmark_Runner SHALL emit a descriptive error message identifying the file path and the reason for failure.

---

### Requirement 6: Comparison Report

**User Story:** As a benchmark operator, I want a comparison report showing per-session reduction ratios and an assessment against the claimed 10–15x reduction, so that I can validate or challenge the Context Lens Power's token reduction claim.

#### Acceptance Criteria

1. WHEN both runs are complete, THE Benchmark_Runner SHALL generate a Comparison_Report showing the usage delta and Reduction_Ratio between the Treatment_Run and the Baseline_Run for each Session and in aggregate.
2. THE Comparison_Report SHALL include, for each Session: the Baseline_Run usage, the Treatment_Run usage, the delta (Treatment minus Baseline), and the Reduction_Ratio (Baseline divided by Treatment).
3. THE Comparison_Report SHALL include aggregate totals and an aggregate Reduction_Ratio across all Sessions.
4. THE Comparison_Report SHALL flag Sessions where Context Lens helped most (cross-file tasks with high Reduction_Ratio) and Sessions where it helped least (single-file tasks with low Reduction_Ratio), using the task type label from the Session_Script.
5. THE Comparison_Report SHALL include an assessment comparing the aggregate Reduction_Ratio against the claimed 10–15x reduction target, stating whether the claim is supported, partially supported, or not supported by the data.
6. THE Benchmark_Runner SHALL print a summary table to standard output showing per-session Reduction_Ratios, deltas, and the aggregate assessment.
7. FOR ALL Sessions in the Comparison_Report, the reported delta SHALL equal the Treatment_Run usage minus the Baseline_Run usage (comparison correctness property).
8. FOR ALL Sessions in the Comparison_Report, the reported Reduction_Ratio SHALL equal the Baseline_Run usage divided by the Treatment_Run usage (reduction ratio correctness property).
9. WHERE the output format is configured as JSON, THE Benchmark_Runner SHALL write the Comparison_Report as a valid JSON document.
10. WHERE the output format is configured as CSV, THE Benchmark_Runner SHALL write the Comparison_Report as a valid CSV document with a header row.

---

### Requirement 7: Run Configuration

**User Story:** As a benchmark operator, I want to control run parameters from a config file — including the path to the target codebase — so that I can reproduce and share exact benchmark conditions.

#### Acceptance Criteria

1. THE Config_File SHALL support the following parameters: `repo_path` (path to the Target_Repository), Prompt_File path, output directory, output format (JSON or CSV), proxy port, and request timeout in seconds.
2. WHEN a required Config_File parameter is missing, THE Benchmark_Runner SHALL emit a descriptive error identifying the missing parameter and halt execution.
3. WHERE a Config_File parameter has a defined default value, THE Benchmark_Runner SHALL apply the default when the parameter is omitted.
4. THE Benchmark_Runner SHALL log the resolved configuration at the start of each run.
5. THE Config_File_Parser SHALL parse a Config_File then serialize it back to an equivalent representation such that re-parsing the serialized form produces an identical configuration (round-trip property).
6. WHEN generating the Session_Script, THE Benchmark_Runner SHALL use the `repo_path` parameter to tailor Session prompts to the structure and content of the Target_Repository.
